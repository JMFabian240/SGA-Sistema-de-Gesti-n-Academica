# RF-40: Registrar alumnos de nuevo ingreso (Wizard)

| Campo | Descripción |
| :---- | :---- |
| **Nombre** | RF-40: Registrar alumnos de nuevo ingreso (Wizard) |
| **Actor** | Administrador o Gestor Administrativo |
| **Objetivo** | Permitir el registro formal y persistencia atómica del expediente de un nuevo estudiante, capturando datos personales, escolares y de seguridad, vinculándolo al perfil del padre o tutor correspondiente y dejándolo disponible para asignación de becas, registro de calificaciones y cobro de colegiaturas. |
| **Flujo Principal** | 1. Actor: Selecciona "Gestión de Alumnos" → "Registrar nuevo alumno" en la interfaz.<br>2. Sistema: Valida token de sesión activo y privilegios del rol (ROLE\_ADMIN / ROLE\_GESTOR).<br>3. Sistema: Renderiza el formulario de registro con todos los campos requeridos.<br>4. Actor: Ingresa los datos personales del alumno: nombre completo, CURP, número de matrícula y fecha de nacimiento.<br>5. Actor: Selecciona el nivel educativo (preescolar, primaria, secundaria o bachillerato), el grado y el grupo del catálogo disponible.<br>6. Actor: Ingresa los nombres de las personas autorizadas para recoger al alumno.<br>7. Actor: Asocia al alumno con un perfil de tutor existente buscándolo por RFC o nombre; o inicia la creación de un nuevo tutor (ver Punto de Extensión RF-11).<br>8. Sistema: Verifica en base de datos que la CURP y la matrícula ingresadas no existan en ningún expediente activo o inactivo.<br>9. Actor: Hace clic en "Guardar expediente".<br>10. Sistema: Valida que todos los campos obligatorios estén completos y con formato correcto.<br>11. Sistema: Inicia transacción SQL; inserta el nuevo expediente vinculado al ID del tutor y ejecuta Commit.<br>12. Sistema: Registra la acción en la bitácora de auditoría con usuario ejecutor, fecha-hora, módulo afectado y tipo de operación.<br>13. Sistema: Muestra mensaje de éxito: "Alumno registrado correctamente. Expediente disponible para asignación de becas y cobros". |
| **Flujo Alterno** | A1. Matrícula o CURP duplicada (Paso 8): 	El sistema detecta la duplicidad, detiene el flujo y muestra: "Error: La CURP/Matrícula ya se encuentra registrada en el sistema". El actor corrige el dato erróneo y regresa al paso correspondiente.   A2. Nivel educativo no pertenece al catálogo (Paso 5): 	El sistema rechaza el valor fuera de catálogo y obliga a seleccionar entre las opciones disponibles: preescolar, primaria, secundaria o bachillerato.   A3. Tutor no vinculado (Paso 7): 	Si el actor intenta guardar sin asociar un tutor, el sistema detiene el flujo y resalta el campo como obligatorio: "Debe vincular un perfil de padre o tutor antes de guardar".   A4. Cancelación (Paso 9): 	El actor hace clic en "Cancelar". El sistema descarta los datos en caché y regresa al listado de alumnos sin persistir ningún registro.   A5. Campos obligatorios incompletos (Paso 10): 	El sistema resalta en rojo los campos faltantes y muestra: "Por favor, complete todos los campos obligatorios antes de continuar". |
| **Precondiciones** | • Sesión activa con token válido y rol ROLE\_ADMIN o ROLE\_GESTOR. • Debe existir al menos un ciclo escolar en estado ACTIVO en el sistema. • El catálogo de niveles educativos, grados y grupos debe estar configurado. |
| **Postcondiciones** | • Expediente del alumno persistido atómicamente en la base de datos con estado ACTIVO. • Expediente vinculado al perfil del tutor correspondiente. • Registro inmutable generado en la bitácora de auditoría (RF-08). • El alumno queda disponible para: asignación de beca (RF-28), registro de calificaciones (Módulo 11\) y generación de cobros de colegiatura (Módulo 6). |
| **Reglas de negocio involucradas** | RN-01: El nivel educativo debe pertenecer estrictamente al catálogo oficial (preescolar, primaria, secundaria o bachillerato). RN-02: CURP y matrícula son identificadores únicos a nivel sistema; no se permiten duplicados ni entre expedientes activos ni inactivos. RN-03: La vinculación con un perfil de padre/tutor es obligatoria para completar el registro. RN-04: La operación de inserción debe ser atómica — un fallo en cualquier paso dispara Rollback completo (sin expedientes parciales). RN-05: La acción queda registrada inmutablemente en la bitácora de auditoría indicando usuario, fecha-hora y módulo afectado. |
