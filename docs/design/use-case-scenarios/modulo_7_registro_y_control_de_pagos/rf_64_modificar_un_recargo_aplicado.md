# RF-64: Modificar un recargo aplicado

| Campo | Descripción |
| :---: | ----- |
| **Autor** | Diana Jareth Ritt Martinez |
| **Nombre** | RF-64: Modificar un recargo aplicado |
| **Actor** | Administrador o Gestor Administrador |
| **Objetivo** | Permitir al personal administrativo modificar el monto de un recargo de $400 aplicado a una colegiatura, registrando el nuevo importe y el motivo del ajuste, para actualizar el adeudo total y conservar evidencia de la modificación en el historial de pagos. |
| **Flujo Principal** | 1. El usuario accede al módulo de pagos.<br>2. Busca y selecciona al alumno.<br>3. El sistema muestra los recargos registrados.<br>4. El usuario selecciona el recargo a modificar.<br>5. El sistema muestra la información del recargo.<br>6. El usuario selecciona **Modificar recargo**.<br>7. Ingresa el nuevo monto y el motivo de la modificación.<br>8. El sistema valida la información.<br>9. El sistema muestra un resumen de los cambios.<br>10. El usuario confirma la operación.<br>11. El sistema actualiza el recargo y recalcula el adeudo.<br>12. El sistema registra la modificación en el historial y muestra un mensaje de éxito. |
| **Flujo Alterno** | A — El usuario no ingresa el motivo<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa el nuevo monto.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El usuario intenta guardar sin registrar el motivo.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema detecta que la justificación está vacía.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema informa que el motivo es obligatorio.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El sistema no guarda la modificación.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El usuario registra el motivo.<br>&nbsp;&nbsp;&nbsp;&nbsp;7. El flujo continúa desde la validación de los datos. B — El nuevo monto no es válido<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa un monto negativo o con un formato incorrecto.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema detecta que el valor no es válido.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema solicita corregir el monto.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema no actualiza el recargo.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El usuario ingresa un valor permitido.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El sistema vuelve a calcular el adeudo. C — El recargo se reduce a cero<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa cero como nuevo monto.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema interpreta la operación como una eliminación o condonación total del recargo.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema solicita confirmar la decisión.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El usuario confirma.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El sistema cambia el recargo a cero.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El sistema recalcula el adeudo.<br>&nbsp;&nbsp;&nbsp;&nbsp;7. El sistema registra que el recargo fue condonado.<br>&nbsp;&nbsp;&nbsp;&nbsp;8. El sistema conserva el motivo y el usuario responsable. D — El recargo ya fue pagado<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario selecciona un recargo.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema detecta que el recargo ya fue cubierto dentro de un pago.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema informa que no puede modificarse directamente sin afectar una operación registrada.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema no cambia el recargo.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El usuario debe seguir el procedimiento autorizado para corregir un pago ya realizado.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El caso de uso termina. E — No existen recargos aplicados<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario consulta el historial del alumno.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema detecta que no hay recargos registrados.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema informa que no existen recargos disponibles para modificar.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema no muestra la opción de modificación.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El caso de uso termina. F — El usuario cancela la modificación<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa el nuevo monto y el motivo.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. Antes de confirmar, selecciona **Cancelar**.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema cierra el proceso.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El recargo conserva el monto anterior.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El adeudo total permanece sin cambios.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El sistema no registra una modificación. G — El recargo ya fue modificado previamente<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario selecciona un recargo que ya fue ajustado.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema muestra el monto original y las modificaciones anteriores.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El usuario registra un nuevo monto y motivo.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema conserva todas las modificaciones en el historial.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El usuario confirma la nueva actualización.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El sistema recalcula el adeudo utilizando el último monto autorizado. |
| **Precondiciones** | El usuario debe haber iniciado sesión. El usuario debe tener rol de Administrador o Gestor Administrativo. El alumno debe estar registrado. Debe existir una colegiatura con recargo aplicado. El recargo debe estar relacionado con un periodo determinado. El usuario debe contar con permisos para modificar recargos. El registro del recargo debe estar disponible para su consulta.   |
| **Postcondiciones** | El recargo queda actualizado. El adeudo total queda recalculado. El monto anterior se conserva en el historial. El nuevo monto y el motivo quedan registrados. La operación identifica al usuario responsable. El historial financiero muestra el ajuste realizado. |
| **Reglas de negocio involucradas** | Solo el Administrador o Gestor Administrativo autorizado puede modificar recargos. El motivo de la modificación es obligatorio. El nuevo monto no puede ser negativo. Un recargo puede reducirse, aumentarse o quedar en cero cuando exista una justificación autorizada. El adeudo debe recalcularse inmediatamente después de la modificación. El valor anterior no debe eliminarse del historial. Toda modificación debe registrar el monto anterior, monto nuevo, motivo, fecha y usuario. Un recargo ya pagado no debe modificarse mediante este procedimiento. La condonación total debe registrarse expresamente. Las modificaciones sucesivas deben conservarse en el historial. |
