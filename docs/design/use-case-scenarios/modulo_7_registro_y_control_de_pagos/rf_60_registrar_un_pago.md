# RF-60: Registrar un pago

| Campo | Descripción |
| :---: | ----- |
| **Autor** | Diana Jareth Ritt Martinez |
| **Nombre** | RF-60: Registrar un pago |
| **Actor** | Administrador o Gestor Administrativo |
| **Objetivo** | Permitir al personal administrativo registrar un pago realizado por el padre de familia o tutor, indicando el concepto, monto, fecha y método de pago, para actualizar automáticamente el saldo y el estado financiero del alumno. |
| **Flujo Principal** | 1. El usuario accede al módulo de pagos.<br>2. Busca y selecciona al alumno.<br>3. El sistema muestra los conceptos de cobro pendientes.<br>4. El usuario selecciona el concepto a pagar.<br>5. El sistema muestra el saldo pendiente.<br>6. El usuario ingresa el monto, fecha y método de pago.<br>7. El sistema valida la información.<br>8. El sistema muestra un resumen de la operación.<br>9. El usuario confirma el registro del pago.<br>10. El sistema registra el pago y actualiza el saldo.<br>11. El sistema actualiza el estado del concepto según corresponda.<br>12. El sistema registra la operación en el historial y muestra un mensaje de éxito. |
| **Flujo Alterno** | Flujo alterno A — El monto recibido es menor al saldo pendiente<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa un monto menor al saldo del concepto.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema valida que el monto sea mayor que cero.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema identifica que el pago no cubre el total.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema muestra que la operación será registrada como un abono.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El usuario confirma el pago.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El sistema registra el abono.<br>&nbsp;&nbsp;&nbsp;&nbsp;7. El sistema descuenta el monto recibido del saldo pendiente.<br>&nbsp;&nbsp;&nbsp;&nbsp;8. El sistema conserva el concepto con estado **Pendiente**.<br>&nbsp;&nbsp;&nbsp;&nbsp;9. El sistema registra el saldo restante.<br>&nbsp;&nbsp;&nbsp;&nbsp;10. El sistema muestra el nuevo saldo al usuario. Flujo alterno B — El monto recibido cubre completamente el saldo<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa un monto igual al saldo pendiente.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema verifica que el pago cubre el total.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El usuario confirma la operación.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema registra el pago.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El sistema deja el saldo en cero.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El sistema cambia el estado del concepto a **Pagado**.<br>&nbsp;&nbsp;&nbsp;&nbsp;7. El sistema registra el movimiento en el historial. Flujo alterno C — El monto ingresado supera el saldo pendiente<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa un monto mayor que el saldo del concepto.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema detecta que el importe excede el saldo.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema informa que el monto no puede aplicarse completamente al concepto seleccionado.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema solicita corregir el monto.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El sistema no registra la operación.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El usuario ingresa un monto igual o menor al saldo.<br>&nbsp;&nbsp;&nbsp;&nbsp;7. El flujo continúa desde la validación de los datos. Flujo alterno D — El pago de colegiatura se registra después del día cinco<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario selecciona una colegiatura.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El usuario ingresa una fecha posterior al quinto día del mes.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema detecta que el pago está fuera del periodo de gracia.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema aplica automáticamente el recargo correspondiente al RF-39.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El sistema muestra el monto de la colegiatura, el recargo y el total.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El usuario revisa la información.<br>&nbsp;&nbsp;&nbsp;&nbsp;7. El usuario confirma el pago.<br>&nbsp;&nbsp;&nbsp;&nbsp;8. El sistema registra la colegiatura y el recargo en el historial. Flujo alterno E — El concepto ya está pagado<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario selecciona un concepto.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema detecta que el concepto ya se encuentra completamente pagado.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema informa que no existe saldo pendiente.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema no permite registrar otro pago para el mismo concepto.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El caso de uso termina sin realizar cambios. Flujo alterno F — No se encuentra al alumno<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario busca al alumno por nombre o matrícula.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema no encuentra coincidencias.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema informa que no se encontraron resultados.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El usuario puede realizar una nueva búsqueda.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El sistema no registra ningún pago. Flujo alterno G — El usuario cancela la operación<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario captura los datos del pago.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. Antes de confirmar, selecciona **Cancelar**.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema cierra el proceso.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema no registra el pago.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El saldo del alumno permanece sin cambios. Flujo alterno H — La fecha de pago no es válida<br>&nbsp;&nbsp;&nbsp;&nbsp;1. El usuario ingresa una fecha de pago no permitida.<br>&nbsp;&nbsp;&nbsp;&nbsp;2. El sistema detecta que la fecha es inválida.<br>&nbsp;&nbsp;&nbsp;&nbsp;3. El sistema informa que debe corregirse.<br>&nbsp;&nbsp;&nbsp;&nbsp;4. El sistema no permite registrar el pago.<br>&nbsp;&nbsp;&nbsp;&nbsp;5. El usuario ingresa una fecha válida.<br>&nbsp;&nbsp;&nbsp;&nbsp;6. El flujo continúa desde la validación de los datos. |
| **Precondiciones** | 1\. 	El Administrador o Gestor Administrativo debe haber iniciado sesión. 2\. 	El alumno debe estar registrado en el sistema. 3\. 	El padre de familia o tutor debe estar relacionado con el alumno. 4\. 	Debe existir al menos un concepto de cobro pendiente. 5\. 	Las tarifas correspondientes al ciclo escolar deben estar configuradas. 6\. 	El usuario debe contar con permisos para registrar pagos. |
| **Postcondiciones** | • El pago queda almacenado. • El saldo del concepto queda actualizado. • El estado del concepto cambia a Pagado cuando el saldo llega a cero. • Los abonos conservan el concepto como Pendiente. • La operación aparece en el historial de pagos. • Se registra el usuario responsable de la operación. |
| **Reglas de negocio involucradas** | • Los tipos de pago permitidos son inscripción, arancel, material, uniforme y colegiatura.<br>• Los métodos de pago permitidos son transferencia, depósito y tarjeta.<br>• El monto recibido debe ser mayor que cero.<br>• Un pago parcial debe registrarse como abono.<br>• Un concepto solo puede marcarse como Pagado cuando su saldo llegue a cero.<br>• No debe permitirse registrar pagos duplicados para conceptos completamente pagados.<br>• La colegiatura pagada después del día cinco puede incluir el recargo definido en el RF-39.<br>• Todo pago debe registrarse en el historial financiero del alumno.<br>• La operación debe identificar al usuario que la realizó.<br>• El saldo debe actualizarse inmediatamente después de confirmar el pago. |
